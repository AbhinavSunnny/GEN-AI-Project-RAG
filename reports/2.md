# Understanding the Evaluation Dataset and Query Generation Process

## Background

The project consists of two major phases:

1. Building and evaluating RAG pipelines
2. Comparing the quality of generated outputs across different retrieval methods and language models

The nine RAG pipelines are:

* Dense GPT4o

* Hybrid GPT4o

* Fusion GPT4o

* Dense Llama3

* Hybrid Llama3

* Fusion Llama3

* Dense Mistral

* Hybrid Mistral

* Fusion Mistral

The final goal is to compare these pipelines using:

* ROUGE-L
* BLEU
* BERTScore
* Precision
* Recall
* MRR
* Latency
* Memory Usage
* Context Relevance

---

# Why We Created an Evaluation Dataset

The original CNN/DailyMail dataset contains:

* Articles
* Human-written summaries

Example:

Article:
A long news article...

Summary:
The president signed a healthcare bill after parliamentary approval.

The dataset does not contain search queries.

However, a Retrieval-Augmented Generation (RAG) system requires:

Query
→ Retrieval
→ Generation

Therefore, an evaluation dataset had to be created.

# How the Query Column Was Generated

The CNN dataset only contains:

* article
* summary

It does not contain user questions.

To simulate retrieval queries, a query generation step was introduced.

The following logic was used:

def create_query(summary):

```
words = str(summary).split()

return " ".join(words[:8])
```

The function takes the first few words of the summary and converts them into a retrieval query.

Example:

Reference Summary:

"The president signed a healthcare bill after extensive debate in parliament."

Generated Query:

"The president signed a healthcare bill"

This query is then stored in the evaluation dataset.

---

# Why the Function Is Not Called Again

Many people expect the function to appear in:

6_3_Dense_GPT4o_Eval.ipynb

However, that is not how the workflow operates.

The function was executed only once during dataset creation.

The generated query values were saved into:

evaluation_dataset.csv

Therefore:

3_0_Create_Evaluation_Set.ipynb

creates:

query
article
reference_summary

and saves them.

Later:

6_3_Dense_GPT4o_Eval.ipynb

simply loads the saved file.

No query generation is required anymore.

# Why We Save Queries Instead of Regenerating Them

There are several advantages.

## Reproducibility

Every pipeline receives the exact same query.

This guarantees fair comparison.

Example:

Dense GPT4o

receives:

"The president signed a healthcare bill"

Hybrid GPT4o

receives:

"The president signed a healthcare bill"

Fusion GPT4o

receives:

"The president signed a healthcare bill"

The experiment remains consistent.

---

## Evaluation Fairness

If queries were generated dynamically inside each notebook, different pipelines could receive slightly different inputs.

This would invalidate the comparison.

Saving the queries beforehand ensures identical inputs for all pipelines.

---

## Faster Execution

Query generation only happens once.

The evaluation notebooks simply read the query column.

This reduces processing overhead.

---

# Why We Need query, reference_summary and generated_answer

These fields are required for final evaluation.

Query:

Used for retrieval.

Reference Summary:

Represents the human-written ground truth.

Generated Answer:

Represents the model output.

The comparison becomes:

Reference Summary

vs

Generated Answer

This enables calculation of:

* ROUGE-L
* BLEU
* BERTScore

---

# Why This Design Was Chosen

The design separates:

Dataset Preparation

from

Pipeline Evaluation

This provides:

* Reproducibility
* Fairness
* Scalability
* Easier debugging
* Consistent benchmarking

The evaluation dataset becomes a fixed benchmark that can be reused across all nine RAG pipelines without modification.

This ensures that any difference in performance comes from the retrieval strategy or language model, rather than differences in the evaluation data itself.
